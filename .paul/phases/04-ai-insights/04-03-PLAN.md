<objective>

## Goal

Build a report generation and multi-channel notification system that sends vendors scheduled digests (daily/weekly) of alerts and key insights via email, SMS, and Slack.

## Purpose

Alert fatigue is real. Vendors don't want 10 emails a day for each anomaly. They need periodic digests that roll up alerts and insights into actionable summaries. This plan adds background job scheduling, digest generation, and multi-channel delivery (SMS/Slack) to complement the real-time dashboard alerts.

## Output

- Report generation engine (daily/weekly digests)
- Multi-channel notification channels (SMS via Twilio, Slack via webhooks)
- Report scheduling preferences UI for vendors
- Supabase scheduled jobs (Edge Functions or background jobs)
- Checkpoint for human verification of report delivery

</objective>

<acceptance_criteria>

## AC-1: Report Generation Engine
```gherkin
Given a vendor has alerts and insights from the past 24 hours (daily) or 7 days (weekly)
When report generation runs (triggered by schedule or manual)
Then generates a report containing:
- Summary of alerts (count, types, severity distribution)
- Top 3 actionable recommendations
- Key metrics (revenue, inventory, channel performance)
- List of high-confidence insights
And report is formatted as HTML + plain text for multi-channel delivery
```

## AC-2: Multi-Channel Notifications
```gherkin
Given a scheduled report is generated
When report is ready to send
Then vendor receives report via selected channels:
- Email (primary, required)
- SMS (optional, if phone configured and enabled)
- Slack (optional, if webhook configured and enabled)
And each channel receives appropriately formatted message
And send failures are logged with retry logic
```

## AC-3: Report Scheduling & Preferences
```gherkin
Given a vendor logged into /notifications-settings page
When vendor configures report preferences
Then vendor can:
- Enable/disable daily or weekly reports
- Select delivery channels (email/SMS/Slack)
- Configure SMS phone number (optional)
- Paste Slack webhook URL (optional)
- View preview of report format
And preferences are persisted to Supabase
And reports auto-generate according to schedule
```

## AC-4: Scheduled Execution
```gherkin
Given reports scheduled for 9 AM daily and 6 AM Monday weekly
When scheduled time arrives
Then Supabase scheduled function triggers automatically
And report is generated and sent within 5 minutes
And execution logged with success/failure status
And vendor receives exactly one report per schedule window (no duplicates)
```

       - ReportType: 'daily' | 'weekly'

       - ReportPreference: { vendorId, reportType, frequency, channels: [email, sms?, slack?] }
    
    2. ReportEngine class (src/lib/report-engine.ts):
       - generateDailyReport(vendorId): Collects alerts + insights from past 24h
       - generateWeeklyReport(vendorId): Collects from past 7 days
       - formatAsHTML(report): Returns HTML email template
       - formatAsText(report): Plain text for SMS fallback
       - formatAsSlack(report): Slack block kit JSON
       
       Content includes:
       - Alert summary: count, severity breakdown (critical/warning/info)
       - Top 3 recommendations (filtered by confidence, sorted by impact)
       - Key metrics: revenue trend, inventory status, channel health
       - Insights: top 5 insights from insight engine
       
       Logic:
       - Query alerts from alert_history where created_at >= cutoff
       - Query insights from insights table
       - Aggregate counts and statistics
       - Rank recommendations by confidence + impact
    
    3. Database schema (supabase/migrations/004_reports_schema.sql):
       CREATE TABLE report_preferences (
         id UUID PRIMARY KEY,
         vendor_id UUID NOT NULL REFERENCES vendors(id),
         report_type VARCHAR (daily | weekly),
         enabled BOOLEAN DEFAULT true,
         channels JSONB ['email', 'sms', 'slack'],
         sms_phone VARCHAR,
         slack_webhook_url VARCHAR,
         created_at TIMESTAMP DEFAULT NOW(),
         updated_at TIMESTAMP DEFAULT NOW(),
         UNIQUE(vendor_id, report_type)
       )
       
       CREATE TABLE report_history (
         id UUID PRIMARY KEY,
         vendor_id UUID NOT NULL REFERENCES vendors(id),
         report_type VARCHAR,
         generated_at TIMESTAMP,
         sent_at TIMESTAMP,
         channels_sent JSONB,
         status VARCHAR (success | failed),
         error_message TEXT
       )
    
    Avoid:
    - Hard-coding thresholds; use alert engine + insight engine definitions
    - Generating reports without checking if preferences exist (graceful skip)
    - Including duplicates in recommendations (de-dup by metric)

       - Create supabase function: send-report-sms
       - Accepts: phone_number, report_text

       - Message: Plain text summary (max 160 chars per SMS, chain messages)
       - Error handling: Log failures, retry on timeout
    
    4. Slack channel (webhook):
       - Create supabase function: send-report-slack
       - Accepts: webhook_url, report_slack_format (blocks)
       - Posts to webhook (no auth needed, webhook includes token)
       - Error handling: Verify webhook on first use, log failures
    
    5. Orchestrator (src/lib/notification-channels.ts):
       - sendReportViaChannels(report, preference): Calls all enabled channels in parallel
       - Returns SendResult[] with success/failure per channel
       - Logs to report_history

    - Exposing API keys in client code (all via edge functions with env vars)
    - Sending unvalidated webhook URLs (validate first)

# Phase 4 Plan 05: Smart Threshold Recommendations

**AI-driven threshold optimization using alert history analysis to reduce false positives and recommend ideal sensitivity settings per vendor.**

## Overview

Vendors struggle to find the right sensitivity settings. Plan 05 analyzes dismissal patterns to auto-recommend thresholds that maximize signal (acted-upon alerts) while minimizing noise (dismissed alerts).

## Architecture

```
Alert History (04-02)
    ↓
Dismissal Tracking (05-01)
    ↓
ThresholdRecommender (05-02) ← Analyzes fatigue metrics
    ↓
Recommendations UI (05-03) ← Displays suggestions with reasoning
    ↓
AlertSensitivity Page (05-04) ← Show recommendations + apply
```

## Task Breakdown

### Task 1: Dismissal Tracking (10min)
- Add `dismissed_at`, `dismissed_by`, `dismissal_reason` to alert_history table
- Update `alertEngine.dismissAlert()` to log dismissal events
- Track dismissal reasons: "false_positive", "already_addressed", "low_priority", "other"

### Task 2: Recommender Engine (15min)
- Implement `ThresholdRecommender` class with history analysis
- Methods: `analyzeAlertHistory()`, `calculateRecommendation()`, `getFatigueMetrics()`
- Fatigue detection: >50% dismiss rate = over-sensitive, suggest increase
- Recommendation window: last 30 days (avoid stale data)

### Task 3: UI Components (15min)
- Create `RecommendationCard` showing suggested threshold + reasoning
- Display confidence score (based on sample size: <10=low, 10-50=medium, >50=high)
- Add "Apply" button to apply recommendation immediately
- Show before/after comparison

### Task 4: Integration (10min)
- Load recommendations on AlertSensitivity page mount
- Display RecommendationCard for each anomaly type with suggestions
- Handle recommendation dismissal ("not helpful" → don't show again)
- Track which recommendations were applied (analytics)

## Acceptance Criteria

- AC-1: Dismissal tracking persists to database with reason
- AC-2: Recommender analyzes 30-day alert history and calculates fatigue metrics
- AC-3: Recommendations shown with confidence score and reasoning explanation
- AC-4: Apply button updates settings and dismisses recommendation card
- AC-5: Recommendations only appear if sufficient data (>10 alerts)

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Recommendations based on insufficient data | Require >10 alerts, show confidence score |
| Over-aggressive auto-tuning | Require user approval (no auto-apply) |
| Stale recommendations | Use 30-day window, refresh on page mount |
| False positive fatigue skews analysis | Track dismissal reasons to filter out noise |

## Success Criteria

✓ All 4 auto-tasks completed
✓ Dismissal tracking persists with reasons
✓ Recommendations generated from alert history
✓ UI shows suggestions with reasoning and confidence
✓ Human checkpoint approves accuracy of recommendations
✓ Build passes, no TypeErrors

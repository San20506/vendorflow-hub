<objective>

## Goal

Build a real-time alert system that detects business anomalies (revenue drops, stockouts, trend shifts) and notifies vendors via dashboard and email, integrating with the Gemini insights engine.

## Purpose

Raw insights are reactionary. Vendors need proactive alerts when something critical changes — a revenue cliff, impending stockout, or demand spike. This plan adds monitoring on top of the insights dashboard, enabling vendors to act before problems compound.

## Output

- Alert engine with anomaly detection logic
- Alert notification system (dashboard + email via Supabase edge functions)
- Confidence-based threshold filtering to prevent alert fatigue
- Real-time alert panel in Insights page
- Checkpoint for human verification of alert logic and email delivery

</objective>

## AC-1: Anomaly Detection Engine
```gherkin
Given a vendor has multi-channel sales data and baseline metrics
When alert engine recalculates metrics every 30 minutes
Then detects anomalies: revenue drop >X%, stockouts imminent, growth trend reversal
```

```gherkin

Then alert is not duplicated (deduplication by type + metric + vendor)

And vendors see single consolidated alert instead of duplicates
```

```gherkin
- Alert title ("Revenue drop detected on Shopify")

- Current metric vs baseline comparison
- Recommended action
- Dismiss button
And panel updates in real-time as new alerts fire
```

## AC-4: Email Notification
```gherkin
Given alert triggered with high confidence
When alert meets email threshold criteria
Then Supabase edge function sends email to vendor with:
- Alert summary and metric context
- Link to /insights for full analysis
- Recommended immediate actions
And email is sent within 2 minutes of alert trigger
```

<task type="auto">
  <name>Build alert engine with anomaly detection logic</name>
  <files>src/lib/alert-engine.ts, src/types/alerts.ts</files>
  <action>
    Create AlertEngine class that:
    - Calculates baseline metrics (30-day rolling average for revenue, growth, inventory)
    - Detects 5 anomaly types:
      1. Revenue anomaly: current < baseline * (1 - threshold%) for a channel
      2. Stockout risk: inventory < reorder_point OR days_to_stockout < 7
      3. Trend reversal: growth changes sign OR drops > X% vs trend
      4. Channel shift: sales concentration shifts unexpectedly (Gini coefficient)
      5. Cost anomaly: unit cost per product rises significantly
    - Returns array of Alert objects: {id, vendorId, type, severity, metric, baseline, current, change%, recommendation, confidence, timestamp}
    - Confidence scoring: uses statistical deviation (sigma) from baseline
    - Example: 2-sigma anomaly → confidence 0.85, 3-sigma → 0.95
    - Avoid: Overfitting to recent spikes; hard-coded thresholds (use env vars)
    
    Type definitions (src/types/alerts.ts):
    type AlertType = 'revenue_drop' | 'stockout_risk' | 'trend_reversal' | 'channel_shift' | 'cost_anomaly'
    type AlertSeverity = 'critical' | 'warning' | 'info'
    interface Alert {
      id: string; vendorId: string; type: AlertType; severity: AlertSeverity
      metric: string; baseline: number; current: number; change: number // % change
      recommendation: string; confidence: number; timestamp: string
      status: 'new' | 'ongoing' | 'resolved'
    }
  </action>
  <verify>
    npm test -- alert-engine.test.ts passes (mock metrics, verify anomalies detected)
    Manually: Create vendor with known metrics, trigger anomaly (set revenue to 50% of baseline), verify alert.type = 'revenue_drop' and confidence > 0.8
  </verify>
  <done>AC-1 satisfied: Anomaly detection engine calculates alerts with confidence scores</done>
</task>

<task type="auto">
  <name>Implement alert deduplication and persistence</name>
  <files>src/lib/alert-engine.ts, supabase/migrations/003_alerts_schema.sql</files>
  <action>
    Add to Supabase schema:
    CREATE TABLE alerts (
      id UUID PRIMARY KEY,
      vendor_id UUID NOT NULL REFERENCES vendors(id),
      alert_type VARCHAR NOT NULL,
      severity VARCHAR NOT NULL,
      metric VARCHAR NOT NULL,
      baseline FLOAT NOT NULL,
      current_value FLOAT NOT NULL,
      confidence FLOAT NOT NULL,
      recommendation TEXT,
      status VARCHAR DEFAULT 'new',
      created_at TIMESTAMP DEFAULT NOW(),
      updated_at TIMESTAMP DEFAULT NOW(),
      dismissed_at TIMESTAMP,
      UNIQUE(vendor_id, alert_type, metric)
    )
    
    Add to AlertEngine:
    - checkDuplicate(vendorId, type, metric) → returns existing alert or null
    - persistAlert(alert) → insert if new, update status='ongoing' if exists
    - dismissAlert(alertId) → set status='resolved', dismissed_at=NOW()
    - getActiveAlerts(vendorId) → returns all alerts where status != 'resolved'
    
    Cooldown: 6 hours before duplicate alert fires again for same metric
    Avoid: Writing duplicate alerts to DB; not updating timestamps
  </action>
  <verify>
    npm test -- alert-engine deduplication passes
    Manually: Trigger same revenue anomaly twice, verify only 1 alert created with updated_at changed
  </verify>
  <done>AC-2 satisfied: Alerts deduplicated and persisted with status tracking</done>
</task>

          <a href="https://app/insights">View in Dashboard</a>
        `
      }
    - Error handling: Log failures, retry up to 2x on timeout
    - Avoid: Sending emails for low-confidence alerts; blocking main flow
    
    Add to alert-engine.ts:
    - shouldNotifyEmail(alert) → returns alert.confidence >= THRESHOLD

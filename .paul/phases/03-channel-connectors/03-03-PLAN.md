<objective>
## Goal

Implement WooCommerce REST API authentication and product sync, enabling vendors to connect their WooCommerce stores and import catalog inventory into VendorFlow.

## Purpose

WooCommerce is the third channel priority (after Shopify and Amazon). Vendors with self-hosted or WooCommerce.com stores need to centralize inventory from their store. This plan reuses the connector framework from Plans 01-02 — only the WooCommerce-specific REST API integration is new.

## Output

- WooCommerce REST API authentication (consumer key + secret, stored securely)
- WooCommerce product catalog fetch (list all products and stock levels)
- WooCommerceConnector class (extends BaseConnector)
- Environment configuration for WooCommerce credentials (.env.local)
- UI integration (WooCommerce button in "Add Channel" dialog, sync handler)
</objective>

## Technology Notes
- WooCommerce uses Basic Auth (Consumer Key:Secret base64-encoded in Authorization header)
- REST API endpoint: GET /wp-json/wc/v3/products
- Pagination: per_page parameter (max 100), supports page numbers
- No token refresh needed (credentials are persistent)
- Simple REST API, no rate limiting documented for small stores

```gherkin
Given WooCommerce is connected and authenticated
When vendor clicks "Sync Now" for WooCommerce channel
Then fetch all products from /wp-json/wc/v3/products with pagination
And normalize to local ChannelProduct format (name, price, stock)
And insert/update in products table with (channel_id, external_product_id)
And handle pagination (WooCommerce default 10, max 100 per page)
```

```gherkin

Then log error per-product (not blocking full sync)
Then retry transient failures (5xx, timeout) with exponential backoff
And update sync status to 'error' with specific failure message
```

**Context:** Plan includes option to implement WooCommerce authentication as either:
1. **Form-based:** Vendor enters store URL + consumer key + secret in a dialog (simple, no OAuth setup needed)
2. **Environment variables:** WooCommerce store URL, key, secret from .env.local (same pattern as Shopify)
3. **Hybrid:** Both form and env vars, with dialog as fallback (most flexible)

**Question:** Which authentication approach should we use?
- **Option 1 (Form-based):** Best UX, works for any WooCommerce store, but requires dialog implementation
- **Option 2 (Env vars only):** Simplest for MVP (reuses Shopify pattern), but less vendor-friendly
- **Option 3 (Hybrid):** Best of both — form input with env var fallback

**Default:** Option 3 (Hybrid) — matches Shopify pattern, maximizes flexibility

<decisions>

**Decision 1: Hybrid Auth (Form + Env Vars)**
- Rationale: Balances vendor UX (form input) with MVP simplicity (env var fallback like Shopify)
- Matches established pattern from Shopify and Amazon plans
- Allows testing with demo WooCommerce store via form input
- Alternative rejected: Pure env vars (less vendor-friendly for MVP demo)

**Decision 2: WooCommerceConnector extends BaseConnector**
- Rationale: Maximizes code reuse (60%+ shared from Shopify/Amazon)
- Only implements getAuthUrl, handleAuthCallback, fetchProducts
- Template method pattern proven effective with Shopify + Amazon
- Alternative rejected: Separate implementation (code duplication)

**Decision 3: REST API pagination (per_page=100)**
- Rationale: Matches WooCommerce REST API defaults, 100 is reasonable page size
- Async generator pattern proven with Shopify/Amazon
- Handles stores with 1000+ products without memory issues
- Alternative rejected: Cursor-based (WooCommerce uses offset pagination only)

**Decision 4: Basic Auth via Authorization header**
- Rationale: Simplest security model (no token refresh, no OAuth setup)
- WooCommerce REST API supports Basic Auth natively
- Credentials stored securely in channels table
- Alternative rejected: OAuth (requires additional WooCommerce app registration)

</decisions>

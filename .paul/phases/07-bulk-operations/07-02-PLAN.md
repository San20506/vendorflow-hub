<objective>

## Goal
Implement soft-delete with 5-level rollback versioning, allowing vendors to recover from accidental bulk deletions and maintain compliance audit trails.

## Purpose
Bulk delete operations can be catastrophic — a vendor might accidentally mark 500 products as deleted. Soft-delete + rollback history (5 states max, FIFO rotation) provides a safety net: vendors can restore from any of the 5 previous versions. This supports post-import corrections without data loss risk.

## Output
- Soft-delete schema: `is_deleted`, `deleted_at` flags on products/orders/settlements
- Rollback versioning table: `deleted_records_history` storing JSON snapshots (5 per record, FIFO)
- Bulk Delete Modal (field selection → preview → soft-delete + versioning)
- Delete History Panel (restore from any of 5 versions, view deletion timeline)
- Edge function for atomic delete + versioning + audit logging
- Audit trail (deletion + restore operations with user context)

</objective>

## AC-3: 5-Level Rollback with FIFO Rotation
```gherkin
Given a record deleted once (version 1 in history)
When vendor deletes same record 5 times total
Then history shows versions 1-5
When vendor deletes a 6th time
Then version 1 (oldest) is purged, versions shift 2→1, 3→2, etc.
And new deletion becomes version 5 (newest)
And vendor can still restore from any of versions 1-5
```

## AC-4: Restore from Rollback State
```gherkin
Given a deleted record with history versions 1-3
When vendor clicks "Restore to Version 2"
Then record restored: is_deleted=false, all fields from version 2 snapshot
And deleted_at cleared
And audit log records: user_id, operation=restore, from_version=2, timestamp
And UI shows: "✓ Record restored from version 2"
```

## AC-5: Audit Trail for Deletions & Restores
```gherkin
Given any delete or restore operation
When operation completes
Then audit_logs records:
  - vendor_id, user_id, operation_type (bulk_delete/restore)
  - entity_type, record_id, version (for restore)
  - record_count (for bulk_delete)
  - timestamp, status (success/failure)
And developer can query: SELECT * FROM audit_logs WHERE operation_type IN ('bulk_delete', 'restore')
```

## AC-6: Error Handling & Duplicate Prevention
```gherkin
Given bulk delete operation fails (network error)
When error occurs
Then vendor sees: "Delete failed. Please try again later."
And dev logs capture: error detail, which records attempted
When vendor retries
Then operation succeeds without creating duplicate history entries
And idempotent: re-running same delete on same record doesn't create extra versions
```

  <name>Implement Bulk Delete Edge Function with Versioning</name>
  <files>supabase/functions/bulk-delete/index.ts</files>
  <action>
    Create edge function for bulk delete with rollback versioning:
    1. Accept: { entityType, recordIds, reason? }
    2. Auth validation (token → user_id, vendor_id from user_profiles)
    3. Fetch current records from table (to snapshot for versioning)
    4. FOR EACH record:
       a. Check deleted_records_history: COUNT versions where record_id = ?
       b. If version_count >= 5: DELETE oldest version (version_number = 1, shift others down)
       c. INSERT new version into deleted_records_history with version_number = 5
       d. UPDATE record SET is_deleted=true, deleted_at=NOW()
    5. Create audit_logs entry: operation_type='bulk_delete', record_count=N, status='success'
    6. Return: { success: bool, deletedCount: number, versionsCreated: number }
    
    Error handling: Vendor sees "Delete failed. Please try again later." (dev logs get full error)
    Idempotence: If record already is_deleted=true, skip updating (don't create duplicate versions)
    
    Avoid: Hard delete (never remove records, only mark is_deleted), not capturing snapshot before delete
  </action>
  <verify>
    1. Build TypeScript: `npm run build` (no errors)
    2. Test function locally (requires Supabase local setup)
    3. Verify audit_logs has entry with operation_type='bulk_delete'
    4. Verify deleted_records_history has snapshot JSON of deleted record
  </verify>

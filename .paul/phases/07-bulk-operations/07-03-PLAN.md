<objective>
## Goal
Implement data export and compliance audit trail functionality for deleted records, enabling vendors to export deletion reports and view complete operation history with configurable export formats.

## Purpose
Compliance and data governance require vendors to maintain exportable records of all delete operations, understand who deleted what and when, and generate deletion reports for audits or data recovery scenarios. This plan extends the soft-delete infrastructure from 07-02 with compliance-focused querying and export capabilities.

## Output
- Export modal for deleted records reports (CSV, JSON, PDF formats)
- Full audit trail page showing all delete/restore operations with filtering
- Optional: deletion reason tracking schema and UI
- Three edge functions: export-deleted-records, fetch-audit-logs, optionally delete-reason-update
- Extended bulk-operations library with export and audit functions
- Audit trails page integrated into main navigation
</objective>

<acceptance_criteria>

## AC-1: Export Deleted Records with Multi-Format Support
```gherkin
Given a vendor has deleted records available in deleted_records_history
When the vendor opens the export modal and selects a format (CSV, JSON, or PDF)
Then the export modal calls the export-deleted-records edge function
And receives a downloadable file containing record details (record_id, entity_type, deleted_at, deleted_by, snapshot summary)
And the export respects vendor_id scoping (only exports their records)
```

## AC-2: Full Audit Trail Query Interface
```gherkin
Given a vendor navigates to the Audit Trails page
When the page loads, it fetches all audit_logs for that vendor from the past year
Then a timeline view displays all delete, restore, and bulk operations with:
  - Operation type (bulk_delete, restore, etc.)
  - Entity type and record count/ID
  - User who performed operation
  - Timestamp (readable format)
  - Metadata (e.g., which version was restored from)
And the vendor can filter by operation type, date range, or entity type
```

## AC-3: Deletion Reason Tracking (Optional)
```gherkin
Given the soft-delete schema includes an optional deletion_reason field
When a vendor deletes records, they can optionally provide a reason (required reason dropdown in delete modal)
Then the reason is stored in audit_logs.metadata or deleted_records_history
And the reason appears in audit trail view and export reports
```

## AC-4: Error Handling and Vendor Scoping
```gherkin
Given an authenticated vendor user calls any export or audit function
When the function receives the auth token and extracts vendor_id
Then:
  - Unauthorized requests (no token) return 401
  - Invalid tokens return 401
  - All queries filter by vendor_id (no cross-vendor leakage)
  - Export/query errors return vendor-friendly messages with dev logging
```

    Build Deno edge function accepting POST request with { entityType, format ('csv'|'json'|'pdf'), dateRange? }.

    - Vendor-friendly error handling: return 400 with "Export failed. Please try again later." for users, console.error for devs
